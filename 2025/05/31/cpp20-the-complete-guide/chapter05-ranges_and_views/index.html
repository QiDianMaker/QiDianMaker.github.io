第五章 范围和视图

自第一个 C++ 标准以来，处理容器和其他序列元素的方法一直是使用迭代器来确定第一个元素（begin）和最后一个元素（end）后面的位置。因此，对范围进行操作的算法通常使用两个参数来处理容器的所有元素，容器提供了 begin() 和 end() 等函数来提供这些参数。

C++20 提供了一种新的方式去处理范围。你可以指定作为一个表示范围（元素序列）整体的对象，并将这个对象传递给算法以作为一个整体处理序列的所有元素。

这种改变听起来很简单，但正如您将看到的，它会产生很多后果。对于调用者和实现者来说，处理算法的方式都发生了巨大的变化。因此，C++20 提供了几个处理范围的新特性：

  • 标准算法的全新实现
  • 若干辅助函数和新的实用工具类型
  • 专门用于处理范围的概念
  • 对视图的特殊处理（创建和使用成本较低的轻量级范围）

本章将介绍范围和视图的基本方面和特点。之后的章节将详细讨论。

5.1.4 哨位

为了处理范围，我们必须引入一个新术语，哨位，它表示范围的结束。

在编程中，哨位是一个特殊的值，它标志着结束或终止。典型的例子有：

  • 用空结束符 ‘\0’ 作为字符序列的结尾（例如，用于字符串字面值）
  • 用 nullptr 标记链表的结束
  • 用 -1 标记非负整数列表的结束

在 ranges 库中，哨位定义范围的结束。在 STL 的传统方法中，哨位将是 end 迭代器，它通常与用于迭代集合的迭代器具有相同的类型。

然而，在 C++20 中，不再需要这样做。

end 迭代器必须与定义范围起始点的迭代器类型相同，并用于遍历元素，这一要求导致了一些缺陷。创建一个 end 迭代器可能是代价高昂的，有时甚至可能是不可能的：

  • 如果要使用C字符串或字符串字面值作为范围，首先必须通过遍历字符来计算 end 迭代器，直到找到 ‘\0’。因此，在使用字符串作为范围之前，我们已经完成了第一次迭代。处理所有的元素需要第二次迭代。
  • 一般来说，如果我们用某个值定义一个范围的结束，则适用此原则。如果需要一个 end 迭代器来处理该范围，首先必须遍历整个范围以找到它的 end。
  • 有时迭代两次（一次查找末尾，一次处理范围的元素）是不可能的。这适用于使用纯输入迭代器的范围，例如使用从其读取的输入流作为范围。为了计算输入的末尾（可能是 EOF），我们必须读取输入。再次读取输入是不可能的，或者会产生不同的值。

end 迭代器作为哨位的泛化解决了这个难题。C++20 的范围支持不同类型的哨位（end 迭代器）。它们可以表示“直到’ \0 ‘”，“直到EOF”，或直到任何其他值。它们甚至可以发出“没有终点”的信号来定义无限的范围和“嘿，迭代器，你自己检查是否有终点。”

注意，在 C++20 之前，我们也可以有这种类型的哨位，但它们必须与迭代器具有相同的类型。输入流迭代器就是一个例子：默认构造的 istream_iterator<> 被用来创建流结束迭代器，这样你就可以用算法处理流中的输入，直到文件结束或出现错误：

因此，通过放宽哨位（end 迭代器）现在必须与迭代迭代器具有相同类型的要求，我们获得了几个好处：

  • 我们可以在开始处理之前跳过查找结束的需要（我们在迭代时处理值并同时查找结束）。
  • 对于 end 迭代器，可以使用禁用导致未定义行为的操作的类型（例如调用 operator *，因为末尾没有值）。在编译时，当试图解引用 end 迭代器时，可以使用它来发出错误信号。
  • 定义 end 迭代器变得更加容易。

5.1.5 用哨位和计数来定义范围

范围不仅可以是容器或一对迭代器。范围还可以通过以下方式定义：

  • 同一类型的开始迭代器和结束迭代器
  • 一个开始迭代器和一个哨位（可能是不同类型的结束标记）
  • 一个开始迭代器和一个计数器
  • 数组

范围支持所有这些范围的定义和使用。

5.1.8 范围的限制和缺点

至少在 C++20 中，范围也有一些主要的限制和缺点，应该在它们的一般介绍中提到：

  • 目前还不支持数值算法的范围。要将范围传递给它们，必须显式地传递 begin() 和 end()：
  • 目前的范围还不支持并行算法。
  • 一些使用 begin 和 end 的传统 API 要求它们具有相同的类型（容器，命名空间std中的算法）。您可能必须与 std::views::common() 或 std::common_iterator 协调它们的类型。
  • 对于某些视图，当视图为 const 时，不能遍历元素。因此，泛型代码可能必须使用通用/转发引用。
  • cbegin() 和 cend() 函数的设计目的是确保您不会（意外地）修改所遍历的元素，但对于引用非 const 对象的视图来说，它们被破坏了。
  • 当视图引用容器时，它们的常量性传播被破坏了。
  • 范围会导致严重的名称空间混乱。考虑下面的 sort() 声明：

此外，在命名空间 std 和命名空间 std::ranges 中有一些符号的行为略有不同。这里，equal_to 就是这样一个例子。您也可以使用 std::equal_to，但在一般实用程序中，std::range 提供了更好的支持，并且对于极端情况更健壮。

5.2 使用视图

如前所述，视图是轻量级范围，可以用作构建块来处理其他范围和视图的所有或部分元素的（修改过的）值。

C++20 提供了几种视图类型和范围适配器来创建它们。几乎每种视图类型都有一个相应的范围适配器，它是一个函数对象，允许我们调用函数来创建视图。适配器定义在命名空间 std::views 中（相当于命名空间 std::ranges::views），并且通常具有不带 _view 后缀的视图名。

范围和视图适配器提供了创建视图的最简单方法。它们可以用来将一个范围转换为一个视图，或者将一个视图转换为一个范围/视图，其中元素可以以各种方式修改：

  • 过滤掉元素
  • 修改元素的值
  • 改变元素的顺序
  • 组合或创建子范围

表格初始视图列出了 C++20 中产生初始视图的标准视图和适配器。它们尤其可以作为视图管道中的起始构建块（要么接受容器等范围作为输入，要么自己生成值）。

std::string_view 是 C++20 之前唯一已经存在的视图类型。所有其他视图都是 C++20 引入的，通常以 _view 结尾(唯一不以 _view 结尾的视图类型是 std::span)。

通常，适配器允许程序员轻松地创建具有最佳性能的视图。为此，适配器创建的视图类型可能取决于它的输入参数。

5.2.1 从范围构建视图

容器和字符串不是视图因为他们不够轻量：它们拥有并复制它们的元素。

要使用容器和字符串作为视图，您应该使用 all() 适配器或已经对范围执行了一些修改的特定视图。

如上所述，你对此可以有不同的选择：

  • 你可以将容器作为参数传递给适配器：

auto first4 = std::views::take(coll, 4);

  • 你可以用管道将容器连接到适配器中：

auto first4 = coll | std::views::take(4);

  • 你可以将容器作为参数传递给视图类型：

std::ranges::take_view first4 {coll, 4};
